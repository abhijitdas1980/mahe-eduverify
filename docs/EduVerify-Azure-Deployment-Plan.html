EduVerify — Azure Deployment Plan

Document: Production deployment architecture for Azure
Application: EduVerify Pre-Verification Portal (Node.js + PostgreSQL)
Scale target: ~5,000 students (admission season), scalable to 25,000+
Prepared: June 2026

Executive summary: Deploy EduVerify as a single Node.js web application on Azure App Service (Linux), with Azure Database for PostgreSQL Flexible Server for all student and verification data. Document files can remain on Cloudinary for the fastest migration, or move to Azure Blob Storage for full in-Azure compliance. This architecture supports real-time orientation-week verification with room to grow.

4. Azure resources checklist

Create these in a dedicated resource group (e.g. rg-eduverify-prod):

  1. Resource Group — rg-eduverify-prod
  2. App Service Plan — Linux, Premium v3
  3. Web App — eduverify-prod
  4. PostgreSQL Flexible Server — eduverify-pg-prod
  5. Key Vault — store JWT, database URL, admin password, Cloudinary keys
  6. Storage Account — optional, for Azure Blob documents
  7. Application Insights — performance and error monitoring
  8. Log Analytics Workspace — centralised logs
  9. Azure Front Door — optional WAF and global access
  10. Private Endpoint — optional, database accessible only inside VNet

5. Security configuration

  • PostgreSQL: Disable public access where possible; use VNet integration or Private Endpoint. Require SSL (sslmode=require).
  • HTTPS: Enforce HTTPS only on App Service; TLS 1.2 minimum.
  • CORS: Set CORS_ORIGIN to exact production URL (never * in production).
  • JWT_SECRET: Minimum 32 random characters; stored in Key Vault.
  • Admin password: Strong SEED_ADMIN_PASSWORD in Key Vault; production syncs to ADM-001 on deploy.
  • Managed Identity: App Service reads secrets from Key Vault without storing them in code.
  • Region: Deploy in Central India or institution-approved Azure region for data residency.

7. CI/CD deployment pipeline

Phase 1 — Manual pilot

  1. Connect GitHub repository to Azure App Service (Deployment Center).
  2. Build: npm ci --omit=dev
  3. Start command: npm start
  4. First boot runs database setup automatically.

Phase 2 — Automated production

Git push (main branch)
    → GitHub Actions or Azure DevOps
    → npm ci
    → Deploy to App Service staging slot
    → Smoke test GET /api/health
    → Swap staging → production (zero-downtime)

8. Migration from Render

StepAction
1Export PostgreSQL from Render using pg_dump
2Create Azure PostgreSQL and import the dump
3Deploy application to App Service with Azure DATABASE_URL
4Keep same Cloudinary account (documents unchanged) or migrate to Blob
5Update DNS to Azure custom domain
6Update CORS_ORIGIN to new URL
7Test full workflow: login, upload, declaration, verification, reassign
8Decommission Render after successful cutover

Planned downtime: 1–4 hours for database cutover, or run parallel systems for one day.

11. Rollout phases

Phase A — Foundation (1–2 weeks)

  • Create Azure subscription, resource group, PostgreSQL, App Service, Key Vault
  • Deploy EduVerify from GitHub; configure SSL and custom domain
  • Set security: JWT, CORS, firewall, strong admin password

Phase B — Data & load test (1 week)

  • Bulk upload 5,000 student records via admin Excel template
  • Load test: concurrent uploads and live verification dashboard
  • Backup and restore drill

Phase C — Go-live (admission season)

  • Staff training (supervisor + verifiers)
  • Monitor Application Insights during orientation week
  • Auto-scale enabled; support runbook ready

Phase D — Hardening (post go-live)

  • Migrate documents to Azure Blob if required by IT policy
  • Private endpoints for database; WAF via Front Door
  • Archive prior-year admission data
