EduVerify — Azure-Only Deployment Plan

Document: Full Microsoft Azure deployment with Azure Blob Storage (no Cloudinary)
Application: EduVerify Pre-Verification Portal · Node.js 20 + PostgreSQL
Scale target: ~5,000 students per admission cycle (scalable to 25,000+)
Prepared: June 2026

Yes — this is possible. EduVerify can run entirely on Azure with student documents stored in Azure Blob Storage instead of Cloudinary. The application already keeps file metadata in PostgreSQL (file_public_id, file_name, file_format) and streams previews through the API. Replacing the Cloudinary module with an Azure Blob adapter is a focused code change (~1–2 days development + testing), not a full rewrite.

3. Why Azure Blob replaces Cloudinary cleanly

Cloudinary is used in four ways in EduVerify. Each maps directly to Azure Blob:

Cloudinary functionUsed forAzure Blob equivalent
uploadBuffer()Student document uploadBlockBlobClient.upload() to path {appNo}/{docCode}_{timestamp}.pdf
signedUrl()Time-limited download linkUser delegation SAS or account SAS (1-hour expiry, read-only)
fetchAssetBuffer()Admin preview + ZIP exportdownloadToBuffer() via SDK (server-side, no public URL)
destroyAsset()Re-upload / student deletedeleteBlob()

The database column file_public_id today stores Cloudinary public IDs. After migration it stores the blob path (e.g. CSE2026001/AADHAAR_1719234567890.pdf). No schema change is required.

Effort note: A new file src/config/blobStorage.js replaces src/config/cloudinary.js. Four files import storage today: student.js, admin.js, docStream.js, deleteStudent.js, plus docs.js for signed URLs. CSP headers in server.js remove Cloudinary domains. The cloudinary npm package is removed; @azure/storage-blob and @azure/identity are added.

4. Azure Blob Storage design

4.1 Storage account settings

SettingValue
PerformanceStandard (documents are not latency-critical)
RedundancyLRS for pilot; ZRS or GRS for production
Access tierHot during admission season; lifecycle rule → Cool after 12 months
Public accessDisabled (container private)
EncryptionMicrosoft-managed keys (or CMK in Key Vault if policy requires)
Soft deleteEnabled (7–14 days) — protects against accidental deletion
VersioningOptional — useful when students re-upload documents
Container nameeduverify-documents
Blob path pattern{appNo}/{docCode}_{unixMs}.{ext}

4.2 Security model (equivalent to Cloudinary “authenticated”)

  • Container has no anonymous read — blobs are never world-readable.
  • Students and staff never receive permanent blob URLs. The API generates short-lived SAS tokens (e.g. 60 minutes) for download links, same as today.
  • Previews in the admin modal use same-origin API streaming (/api/.../preview) — already implemented; only the backend fetch source changes from Cloudinary to Blob.
  • App Service uses Managed Identity with role Storage Blob Data Contributor on the container — no account keys in environment variables.
  • Network: optional Private Endpoint for Storage so blob traffic stays on Azure backbone.

4.3 Environment variables (Azure-only — no Cloudinary)

VariableExample / purpose
STORAGE_PROVIDERazure (or remove Cloudinary vars entirely)
AZURE_STORAGE_ACCOUNT_NAMEsteduverifyprod
AZURE_STORAGE_CONTAINEReduverify-documents
AZURE_STORAGE_SAS_EXPIRY_MINUTES60 (download link lifetime)
AZURE_CLIENT_IDOnly if using user-assigned managed identity
Remove: CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET

With Managed Identity, DefaultAzureCredential authenticates automatically on App Service — no storage account key needed.

5. Code migration outline

5.1 New storage module (pseudocode)

src/config/blobStorage.js
  - isConfigured()     → checks AZURE_STORAGE_ACCOUNT_NAME + container
  - uploadBuffer(buf, appNo, docCode) → returns { publicId: blobPath, format, resourceType }
  - signedUrl(doc)     → generateBlobSASUrl(blobPath, read, 60 min)
  - fetchAssetBuffer(doc) → download blob to Buffer
  - destroyAsset(publicId) → deleteBlob(publicId)

src/config/storage.js (thin facade)
  - exports functions from blobStorage OR cloudinary based on STORAGE_PROVIDER
  - allows phased cutover during migration

9. Migration from Render + Cloudinary

StepAction
1Provision Azure resources (App Service, PostgreSQL, Storage, Key Vault)
2Deploy EduVerify with Blob storage code + Managed Identity
3pg_dump from Render PostgreSQL → restore to Azure PostgreSQL
4Run migration script: for each document with Cloudinary file_public_id, download via Cloudinary API and upload to Blob; update file_public_id to blob path
5Verify sample students: preview, download, ZIP export, re-upload, delete
6Point DNS to Azure; set CORS_ORIGIN to new URL
7Decommission Render and Cloudinary after 30-day parallel run (optional)

Greenfield deploy (no existing uploads): Skip step 4 — deploy Blob code directly; no Cloudinary account needed.

10. Security & compliance

  • All student PII and documents remain in chosen Azure region (data residency).
  • PostgreSQL: SSL, firewall / VNet, least-privilege DB user.
  • Blobs: private container, encryption at rest, soft delete, audit logs via Storage Analytics.
  • No third-party document CDN — reduces vendor risk and DPA scope.
  • Key Vault for secrets; Managed Identity eliminates storage keys in config.
  • Existing app controls retained: JWT auth, rate limits, staff roles, audit log.
